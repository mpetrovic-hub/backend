# Dimoco General API

This document summarizes the generic DIMOCO pay:smart API for use in this repository.

It is intentionally **not country-specific** and **not flow-specific**.  
Use this file for everything that is generally true for DIMOCO pay:smart across actions, countries, and integrations.

For country- or setup-specific behavior, add separate files such as:
- `at/<flow>/README.md`
- `pl/<flow>/README.md`
- `de/<flow>/README.md`

Primary source:
- `source/DIMOCO-specification_paysmart.pdf`

## Product overview

DIMOCO pay:smart is a payment and subscription interface that lets a merchant trigger payment-related actions and receive the results asynchronously. It supports actions such as identification, operator lookup, one-off payments, subscription lifecycle management, status retrieval, refunds, and blocklist management. The specification is designed so that DIMOCO handles many flow decisions and operator-specific details on its side. :contentReference[oaicite:0]{index=0}

## Terminology

Important generic terms used by DIMOCO:

- **enduser**  
  The consumer buying or subscribing.

- **merchant**  
  The service provider using pay:smart.

- **MNO**  
  Mobile network operator.

- **PSP**  
  Payment service provider.

- **MSISDN**  
  Subscriber phone number.

- **request_id**  
  Short-lived unique request token generated by the merchant.

- **reference**  
  Short-lived correlation identifier generated by DIMOCO for a single pay:smart request.

- **transaction**  
  Long-lived identifier for a payment or related action.

- **subscription**  
  Long-lived identifier for a subscription.

- **kyc-check**  
  Customer identity matching against operator-held data.

## Service setup

DIMOCO service setup is done with DIMOCO directly. During setup, the merchant receives:
- a `merchant` id
- a password / shared secret used for digest calculation
- one or more `order` ids as service identifiers

Many optional parameters may also be preconfigured on DIMOCO side at merchant or order level. :contentReference[oaicite:1]{index=1}

## Communication model

The generic communication pattern is:

1. The merchant sends an action request to pay:smart.
2. DIMOCO responds synchronously with an XML response.
3. The response may indicate:
   - failure
   - validation failure
   - redirect required
   - pending
4. If the flow continues asynchronously, DIMOCO later sends a callback to the merchant server.
5. If the enduser was redirected to DIMOCO, DIMOCO normally sends the user back after the first callback was successfully delivered. :contentReference[oaicite:2]{index=2}

## Transport variants

### 1. Server-to-server

This is the recommended integration model.

Characteristics:
- most reliable
- more secure
- suitable for the broadest range of flows
- supports retries on callbacks
- keeps payment details away from the enduser browser

### 2. Enduser transport

In this mode, the enduser browser transports request or result data directly.

Characteristics:
- simpler in some cases
- may reduce some latency
- comes with major downsides:
  - not suitable for all flows
  - failed requests may be hard for the merchant to detect
  - no reliable retry semantics
  - action data can become visible to the enduser
  - redirects may require POST rather than simple GET

DIMOCO explicitly recommends server-to-server communication as the default. :contentReference[oaicite:3]{index=3}

## API endpoints

Generic endpoints in the specification:

- Server-to-server API: `https://services.dimoco.at/smart/payment`
- Enduser transport API: `https://services.dimoco.at/smart/userpayment`

Callback security notes:
- callback delivery requires HTTPS
- the merchant server should allow inbound connections from `91.198.93.0/24`
- a valid signed TLS certificate is required on the merchant side :contentReference[oaicite:4]{index=4}

## Request format

Requests to pay:smart use:

- `POST`
- content type: `application/x-www-form-urlencoded`
- UTF-8 string encoding

Parameters are transmitted URL-encoded over the wire. :contentReference[oaicite:5]{index=5}

## Callback format

Callbacks from DIMOCO to the merchant are also sent via:

- `POST`
- content type: `application/x-www-form-urlencoded`
- UTF-8 string encoding

Generic callback fields:
- `data` = XML result document
- `digest` = HMAC-based authenticity check

A callback counts as successfully delivered only if the merchant responds with HTTP `200`. Otherwise DIMOCO retries delivery automatically. :contentReference[oaicite:6]{index=6}

## Enduser return

If an enduser was redirected to DIMOCO, DIMOCO can send the enduser back to the merchant via:
- a simple GET redirect after callback completion
- a POST-based return carrying result data
- enriched return parameters in some configured cases

### Status with return

DIMOCO can enrich `url_return` with:
- `sph-x` = final outcome indicator
- `sph-r` = reference
- `sph-s` = subscription id
- `sph-d` = digest over the transported return values

These values must be validated on merchant side because they are easy to forge if treated as trusted browser data. :contentReference[oaicite:7]{index=7}

### Data with return

For `identify`, DIMOCO can optionally return the final XML result in encrypted form via:
- `sph-e`
- `sph-n`

This flow uses AES-GCM-based decryption and gzip decompression with a key derived from the merchant password. It is an optimization for identify and replaces the usual server-to-server callback in that special case. :contentReference[oaicite:8]{index=8}

## Authentication and digest

The `digest` is mandatory for all pay:smart requests.

Formula:

`hex(hmac_sha256_digest(password, payload))`

### Request digest calculation

For requests **to** DIMOCO:

1. Take all request parameters except `digest`.
2. Sort parameters alphabetically by parameter name.
3. Concatenate **only the unencoded parameter values**.
4. Encode the resulting string as UTF-8.
5. Compute HMAC-SHA256 using the merchant password.
6. Hex-encode the result.

Important rules:
- sign **values only**, not keys
- use **alphabetical order of parameter names**
- use **unencoded** values for the payload
- use UTF-8 consistently

### Callback digest calculation

For callbacks **from** DIMOCO:

- the payload is the full URL-decoded XML result document from the `data` field
- the received `digest` must be recomputed and verified before trusting the callback

Important verification notes:
- verify every callback
- verify every notification
- do not accidentally alter the callback XML before verification
- be careful with trailing characters
- XML values may be escaped inside the XML itself, which is separate from URL decoding

### Callback service resolution in this backend

This backend resolves the internal DIMOCO service in two steps:

1. first by `payment_parameters/order` if present
2. if `order` is missing, by digest verification against all configured DIMOCO service secrets

Safety rule for step 2:
- accept the callback only when exactly one configured service secret matches
- reject the callback when there is no match
- when multiple services match:
  - accept and store the callback if all matched services share the same secret
  - keep service attribution unresolved in that case
  - reject only if multiple matches are present and they do not share one secret

## Repository note

Digest generation and verification are already implemented in the current codebase.  
Do not create alternative digest implementations unless the existing logic is being deliberately refactored.

## Common synchronous result patterns

Most synchronous action responses return at least:
- `action_result/code`
- `action_result/detail`
- `action_result/status`
- `reference`
- `request_id`

Common synchronous status meanings:
- `1` = failure
- `3` = redirect required
- `4` = validation failed
- `5` = pending, final result will follow later

Some actions can return full success synchronously, but many important flows rely on callbacks. :contentReference[oaicite:9]{index=9}

## Common callback result patterns

Final callback XML commonly includes:
- `action`
- `action_result`
- `reference`
- `request_id`
- custom parameters if provided
- customer fields if relevant
- payment fields if relevant
- subscription fields if relevant
- transaction fields if relevant

Common final callback status meanings:
- `0` = success
- `1` = failure

Some actions use additional status variants such as an active subscription without initial billing. :contentReference[oaicite:10]{index=10}

# Notifications for managed resources

DIMOCO can send notifications for events that were not directly triggered by a preceding merchant API call.

Key points:
- notifications can happen without a prior request from the merchant
- they must always be authenticated via digest verification
- they commonly carry `notification = true` in additional results

Documented notification types:
- `start-subscription`
- `renew-subscription`
- `close-subscription`
- `start`
- `receive-sms-info`
- `refund`

Special case `receive-sms-info` can include:
- SMS reference
- SMS service number
- received SMS text
- enduser country
- MSISDN
- operator
- order

## Important rule

Treat notifications with the same authenticity requirements as normal callbacks.  
Never trust them without digest verification.

# Advanced flows

The specification includes advanced flows, but explicitly says that most integrations do not need them unless DIMOCO requires them for a specific setup.

## 1. Early enduser return

Allows the enduser to return early while the final result is still pending.

Interim callback includes:
- `activity_required = true`
- `activity = delay-enduser`

## 2. SMS from enduser

The merchant portal itself asks the user to send an SMS.

Interim callback can include:
- `activity = request-sms`
- SMS text
- service number

## 3. TAN entry on merchant portal

The merchant portal collects the TAN and submits it back to DIMOCO.

Interim callback can include:
- `activity = request-tan`
- `response_url`
- transaction id

Follow-up request contains:
- `result`
- `tan`
- `transaction`
- `digest`

Response format for the follow-up is JSON with an `accepted` field.

## 4. On-demand content provisioning

Content is only generated once payment is ready for finalization.

Interim callback can include:
- `activity = provide-content`
- `response_url`
- transaction id

Follow-up request contains:
- `result`
- `prompt_content_args`
- `transaction`
- `digest`

## 5. Content acknowledgement

Merchant delivers content by own means and later confirms delivery to DIMOCO.

Interim callback can include:
- `activity = ship-content`
- `response_url`
- transaction id

Follow-up request contains:
- `result`
- `transaction`
- `digest`

## 6. Proportional capture of reserved amount

Used when the maximum amount is reserved first and only the consumed amount is captured later.

Interim callback can include:
- `activity = capture-amount`
- `response_url`
- transaction id

Follow-up request contains:
- `result`
- `amount`
- `transaction`
- `digest`

If the actual billed amount is lower than the reserved amount, the callback may include `proportional_capture = true`. :contentReference[oaicite:12]{index=12}

# Action result codes

The specification defines numeric action result codes and notes that more may be added later.

## Request and validation related

- `100` request contains no valid data
- `101` parsing or validation failed
- `102` invalid or missing parameter value
- `103` formally required parameter missing
- `111` merchant unauthorized
- `125` access denied
- `128` too many devices
- `129` user on blocklist
- `141` invalid PIN
- `144` duplicate or illegal concurrent request
- `153` inactive order

## Generic action / internal errors

- `206` PSP limit or insufficient coverage
- `300` internal implementation error
- `301` action invalid for current transaction state
- `302` redirect required
- `304` load exceeded
- `401` invalid amount
- `404` amount not allowed

## PSP and customer related

- `500` PSP unavailable
- `501` PSP communication failed
- `502` PSP configuration invalid
- `503` other PSP error
- `504` PSP load exceeded
- `510` customer unknown
- `512` goods not allowed
- `515` customer cancelled
- `516` subscription error
- `517` service unknown
- `518` concurrent authorization issue
- `519` customer timeout
- `520` payment not activated
- `521` subscription closed
- `522` no balance
- `523` already subscribed
- `525` unsuccessful for non-specific but harmless reason
- `526` fraud detected
- `527` KYC mismatch
- `528` reservation error
- `529` MVNO not supported
- `531` prepaid not allowed
- `535` partial billing
- `540` age verification failure
- `550` SMS delivery failed

## Subscription limits

- `704` PSP-specific subscription limit reached
- `705` subscription already charged for current period

## Blocklist-specific

- `801` entry already exists on requested scope
- `802` entry not found on requested scope

# Integration guidance for this repository

When working with Dimoco in this repository:

- keep raw Dimoco request and callback structures inside the Dimoco integration layer
- keep digest generation and verification centralized
- map XML structures to internal models close to the integration boundary
- do not leak Dimoco-specific field names into shared domain logic
- treat callbacks and notifications as untrusted until digest verification succeeds
- do not assume synchronous success for actions that are specified as pending + callback driven
- do not assume country-specific behavior unless documented in a country/flow file

## Current repository relevance

Actions especially relevant to the current codebase include:
- `operator-lookup`
- `refund`
- `add-blocklist`

But this file intentionally documents the **full generic pay:smart API surface** from the uploaded specification so future extensions can reuse it.
